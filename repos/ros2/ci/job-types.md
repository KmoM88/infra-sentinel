# ROS 2 CI Job Types

This document provides an overview of the different types of Jenkins jobs that are generated by the `ros2/ci` repository. These jobs are categorized by their prefixes, which indicate their purpose and trigger mechanism.

## Job Categories

### `ci_*`

*   **Purpose:** These are the main Continuous Integration (CI) jobs. They are designed to be manually triggered by developers to test their changes across a variety of platforms (Linux, Windows, macOS, etc.).
*   **Trigger:** Manual.
*   **Key Characteristics:**
    *   Build and run tests for a given set of ROS 2 repositories.
    *   Use a `None` build type by default, allowing developers to specify the build type as a parameter.
    *   Spin up a new Docker image for each build to ensure a clean and consistent environment.
    *   The `ci_launcher` job is a special job in this category that triggers all other `ci_*` jobs.

### `ci_packaging_*`

*   **Purpose:** These jobs are used to manually test the packaging process for ROS 2. They build and create packages (Debian or Chocolatey) for the ROS 2 packages in the workspace.
*   **Trigger:** Manual.
*   **Key Characteristics:**
    *   Similar to `ci_*` jobs, but they also perform the packaging step.
    *   Use a `RelWithDebInfo` build type by default.
    *   Archive the generated packages as build artifacts.
    *   Spin up a new Docker image for each build.

### `packaging_*`

*   **Purpose:** These are the automated, nightly versions of the `ci_packaging_*` jobs. They are responsible for generating the official binary packages for ROS 2.
*   **Trigger:** Nightly (time-based).
*   **Key Characteristics:**
    *   Identical to `ci_packaging_*` jobs in terms of functionality.
    *   Run on a fixed schedule.
    *   Send email notifications for build results.

### `nightly_*`

*   **Purpose:** This is a broad category of jobs that run on a nightly basis to perform various types of testing that are too time-consuming to run on every commit.
*   **Trigger:** Nightly (time-based).
*   **Key Characteristics:**
    *   **`nightly_*_debug`:** Run a full build and test cycle with a `Debug` build type.
    *   **`nightly_*_release`:** Run a full build and test cycle with a `Release` build type.
    *   **`nightly_*_repeated`:** Run tests repeatedly until a failure occurs to identify flaky tests.
    *   **`nightly_*_xfail`:** Run only tests that are expected to fail (`xfail`) to monitor the status of known issues.
    *   **`nightly_*_coverage`:** Generate code coverage reports.
    *   **`nightly_*_address_sanitizer`:** Run tests with the Address Sanitizer enabled to detect memory errors.
    *   **`nightly_*_clang_libcxx`:** Build and test with the `clang` compiler and `libc++` standard library.
    *   Spin up a new Docker image for each build.

### `test_ci_*`

*   **Purpose:** These are sandbox jobs for developers to experiment with changes to the CI configuration without affecting the main `ci_*` jobs.
*   **Trigger:** Manual.
*   **Key Characteristics:**
    *   Identical to `ci_*` jobs in terms of functionality.
    *   Do not have email notifications configured.
    *   Provide a safe environment for testing modifications to the build and test process.

### `test_packaging_*`

*   **Purpose:** These are sandbox jobs for developers to experiment with changes to the packaging configuration without affecting the main `packaging_*` jobs.
*   **Trigger:** Manual.
*   **Key Characteristics:**
    *   Identical to `ci_packaging_*` jobs in terms of functionality.
    *   Do not have email notifications configured.
    *   Provide a safe environment for testing modifications to the packaging process.
