# nightly_linux-rhel_xfail Job Analysis

## Overview

The `nightly_linux-rhel_xfail` job is a nightly-triggered Jenkins pipeline for building and testing ROS 2 on a Red Hat Enterprise Linux (RHEL) like environment, with a specific focus on running tests that are expected to fail (`xfail`). It is dynamically generated by the `create_jenkins_job.py` script and uses the `ci_job.xml.em` template.

## Job Generation

The `create_jenkins_job.py` script generates the `nightly_linux-rhel_xfail` job for the `linux-rhel` OS. It's designed to monitor the status of known failing tests on a nightly basis for RHEL.

## XFail Testing-Specific Configuration

The `nightly_linux-rhel_xfail` job has the following key configuration settings:

*   **Job Name:** The job is explicitly named `nightly_linux-rhel_xfail`.
*   **Template:** It uses the `ci_job.xml.em` template.
*   **Jenkins Node Label:** It is configured to run on a Jenkins node with the `linux` label.
*   **Build Type:** The `cmake_build_type` is set to `None`.
*   **Trigger:** The job is triggered by a timer (`time_trigger_spec`) set to run daily at 4:00 AM (`0 4 * * *`).
*   **Notifications:** Email notifications for build results are sent to `ros2-buildfarm@googlegroups.com`.
*   **Test Arguments:** The test arguments are significantly modified to only run `xfail` tests:
    *   `--ctest-args -L xfail`: This flag tells `ctest` to only run tests with the `xfail` label.
    *   `--pytest-args -m xfail --runxfail`: This tells `pytest` to only run tests marked with `xfail` and to run them even though they are expected to fail.
*   **Connext:** `use_connext_debs_default` is set to `false`.
*   **Build Arguments:** The job passes an additional CMake argument: `-DPython3_EXECUTABLE=/usr/bin/python3`.
*   **Docker Image:** The Docker image is tagged as `ros2_batch_ci_rhel`.

## Execution Flow

The execution flow of the `nightly_linux-rhel_xfail` job is similar to the `ci_linux-rhel` job, with the main difference being the set of tests that are executed.

1.  **SCM Checkout:** The job checks out the `ros2/ci` repository.

2.  **Docker Environment:** The job runs inside a Docker container.

    *   **Dockerfile Selection:** The job uses the `linux_docker_resources/Dockerfile-RHEL`.
    *   **Docker Image Build:** A new Docker image is built from `Dockerfile-RHEL` and tagged as `ros2_batch_ci_rhel`.
    *   **Docker Container Run:** The job runs the newly built Docker image, passing the specific build and test arguments via the `CI_ARGS` environment variable.

3.  **`run_ros2_batch.py`:** The `ros2_batch_job` Python module within the container handles the build and test process. The specific `ctest` and `pytest` arguments are passed to `colcon test`.

## Configuration and Scripts

*   **Job Generation Script:**
    *   `create_jenkins_job.py`: The script that generates the `nightly_linux-rhel_xfail` job.
*   **Jenkins Job Template:**
    *   `job_templates/ci_job.xml.em`: The template for the CI job.
*   **Build Logic:**
    *   `run_ros2_batch.py`: The entry point for the build process inside the Docker container.
*   **Docker configuration:**
    *   `linux_docker_resources/Dockerfile-RHEL`: The Dockerfile for RHEL builds.

## Parameters

The `nightly_linux-rhel_xfail` job accepts the same parameters as the `ci_linux-rhel` job, but it is typically run with the default values since it is triggered automatically. The `CI_EL_RELEASE` parameter is used to select the specific RHEL/AlmaLinux version for the build.
