# nightly_linux-aarch64_xfail Job Analysis

## Overview

The `nightly_linux-aarch64_xfail` job is a nightly-triggered Jenkins pipeline for building and testing ROS 2 on the `aarch64` (ARM64) architecture, with a specific focus on running tests that are expected to fail (`xfail`). It is dynamically generated by the `create_jenkins_job.py` script and uses the `ci_job.xml.em` template.

## Job Generation

The `create_jenkins_job.py` script generates the `nightly_linux-aarch64_xfail` job for the `linux-aarch64` OS. It's designed to monitor the status of known failing tests on a nightly basis for the `aarch64` architecture.

## XFail Testing-Specific Configuration

The `nightly_linux-aarch64_xfail` job has the following key configuration settings:

*   **Job Name:** The job is explicitly named `nightly_linux-aarch64_xfail`.
*   **Template:** It uses the `ci_job.xml.em` template.
*   **Jenkins Node Label:** It is configured to run on a Jenkins node with the `linux_aarch64` label.
*   **Build Type:** The `cmake_build_type` is set to `None`.
*   **Trigger:** The job is triggered by a timer (`time_trigger_spec`) set to run daily at 4:00 AM (`0 4 * * *`).
*   **Notifications:** Email notifications for build results are sent to `ros2-buildfarm@googlegroups.com`.
*   **Test Arguments:** The test arguments are significantly modified to only run `xfail` tests:
    *   `--ctest-args -L xfail`: This flag tells `ctest` to only run tests with the `xfail` label.
    *   `--pytest-args -m xfail --runxfail`: This tells `pytest` to only run tests marked with `xfail` and to run them even though they are expected to fail.
*   **RMW Implementation:** It adds `rmw_connextdds` to the default set of ignored RMW implementations.
*   **Docker Build Argument:** It passes the build argument `PLATFORM=aarch64` to the `docker build` command.
*   **Docker Image:** The Docker image is tagged as `ros2_batch_ci_aarch64`.

## Execution Flow

The execution flow of the `nightly_linux-aarch64_xfail` job is similar to the `ci_linux-aarch64` job, with the main difference being the set of tests that are executed.

1.  **SCM Checkout:** The job checks out the `ros2/ci` repository.

2.  **Docker Environment:** The job runs inside a Docker container.

    *   **Dockerfile Selection:** The job uses the standard `linux_docker_resources/Dockerfile`.
    *   **Docker Image Build:** A new Docker image is built from `Dockerfile` and tagged as `ros2_batch_ci_aarch64`.
    *   **Docker Container Run:** The job runs the newly built Docker image, passing the specific build and test arguments via the `CI_ARGS` environment variable.

3.  **`run_ros2_batch.py`:** The `ros2_batch_job` Python module within the container handles the build and test process. The specific `ctest` and `pytest` arguments are passed to `colcon test`.

## Configuration and Scripts

*   **Job Generation Script:**
    *   `create_jenkins_job.py`: The script that generates the `nightly_linux-aarch64_xfail` job.
*   **Jenkins Job Template:**
    *   `job_templates/ci_job.xml.em`: The template for the CI job.
*   **Build Logic:**
    *   `run_ros2_batch.py`: The entry point for the build process inside the Docker container.
*   **Docker configuration:**
    *   `linux_docker_resources/Dockerfile`: The Dockerfile for Linux builds.

## Parameters

The `nightly_linux-aarch64_xfail` job accepts the same parameters as the `ci_linux-aarch64` job, but it is typically run with the default values since it is triggered automatically.
