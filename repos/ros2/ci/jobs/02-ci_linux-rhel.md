# ci_linux-rhel Job Analysis

## Overview

The `ci_linux-rhel` job is a manually-triggered Jenkins pipeline for running a ROS 2 build on a Red Hat Enterprise Linux (RHEL) like environment. Like the `ci_linux` job, it is dynamically generated by the `create_jenkins_job.py` script and is an instance of the generic `ci_job` template.

## Job Generation

The `create_jenkins_job.py` script generates the `ci_linux-rhel` job based on the `linux-rhel` entry in the `os_configs` dictionary. This configuration specifies RHEL-specific settings and points to a different Dockerfile.

## RHEL-Specific Configuration

The `ci_linux-rhel` job has the following key differences from the standard `ci_linux` job:

*   **Dockerfile:** It uses a dedicated Dockerfile located at `linux_docker_resources/Dockerfile-RHEL`. This Dockerfile is based on `almalinux` (a RHEL derivative) and uses `dnf` as the package manager.
*   **Docker Image Tag:** The generated Docker image is tagged as `ros2_batch_ci_rhel`.
*   **CMake Arguments:** The job passes an additional CMake argument: `-DPython3_EXECUTABLE=/usr/bin/python3`.
*   **Connext Debs:** The `use_connext_debs_default` parameter is set to `false`, meaning the job will not attempt to install Connext from Debian packages.

## Execution Flow

The execution flow of the `ci_linux-rhel` job is very similar to the `ci_linux` job:

1.  **SCM Checkout:** The job checks out the `ros2/ci` repository.

2.  **Docker Environment:** The job runs inside a Docker container.

    *   **Dockerfile Selection:** The job selects the `linux_docker_resources/Dockerfile-RHEL`.
    *   **Dockerfile Modification:** The Dockerfile is dynamically modified to use the AlmaLinux release specified by the `CI_EL_RELEASE` parameter.
    *   **Docker Image Build:** A new Docker image is built from `Dockerfile-RHEL` and tagged as `ros2_batch_ci_rhel`.
    *   **Docker Container Run:** The job runs the newly built Docker image, passing the build arguments via the `CI_ARGS` environment variable.

3.  **`run_ros2_batch.py` and `ros2_batch_job`:** The logic inside the container is the same as other `ci` jobs, orchestrated by the `ros2_batch_job` Python module. This script handles repository fetching, building, and testing using `vcstool` and `colcon`.

## Configuration and Scripts

*   **Job Generation Script:**
    *   `create_jenkins_job.py`: The script that generates the `ci_linux-rhel` job.
*   **Jenkins Job Template:**
    *   `job_templates/ci_job.xml.em`: The template for the `ci_job`.
*   **Build Logic:**
    *   `run_ros2_batch.py`: The entry point for the build process inside the Docker container.
    *   `ros2_batch_job/`: The Python module containing the core build and test logic.
*   **Docker configuration:**
    *   `linux_docker_resources/Dockerfile-RHEL`: The Dockerfile for RHEL builds.

## Parameters

The `ci_linux-rhel` job accepts the same parameters as the `ci_launcher` and its other downstream jobs. The `CI_EL_RELEASE` parameter is used to select the specific RHEL/AlmaLinux version for the build. For a full list of parameters, please refer to the documentation for the `ci_launcher` job.
