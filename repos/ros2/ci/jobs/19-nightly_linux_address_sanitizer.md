# nightly_linux_address_sanitizer Job Analysis

## Overview

The `nightly_linux_address_sanitizer` job is a nightly-triggered Jenkins pipeline for building and testing ROS 2 on Linux with the Address Sanitizer (ASan) enabled. ASan is a memory error detector for C/C++. This job is dynamically generated by the `create_jenkins_job.py` script and uses the `ci_job.xml.em` template.

## Job Generation

The `create_jenkins_job.py` script generates the `nightly_linux_address_sanitizer` job for the `linux` OS. It's designed to detect memory errors in a subset of the ROS 2 core packages on a nightly basis.

## Address Sanitizer-Specific Configuration

The `nightly_linux_address_sanitizer` job has the following key configuration settings:

*   **Job Name:** The job is explicitly named `nightly_linux_address_sanitizer`.
*   **Template:** It uses the `ci_job.xml.em` template.
*   **Jenkins Node Label:** It is configured to run on a Jenkins node with the `linux` label.
*   **Build Type:** The `cmake_build_type` is set to `Debug`.
*   **Trigger:** The job is triggered by a timer (`time_trigger_spec`) set to run daily at 4:00 AM (`0 4 * * *`).
*   **Notifications:** Email notifications for build results are sent to `ros2-buildfarm@googlegroups.com` and `ros-contributions@amazon.com`.
*   **Build Arguments:** The build arguments are heavily modified to enable ASan:
    *   `--cmake-args -DOSRF_TESTING_TOOLS_CPP_DISABLE_MEMORY_TOOLS=ON`: This flag disables some of the default memory tools in `osrf_testing_tools_cpp` to avoid conflicts with ASan.
    *   `--mixin asan-gcc`: This `colcon` mixin provides the necessary compiler and linker flags to enable ASan.
    *   `--packages-up-to rcpputils`: This limits the build to `rcpputils` and its dependencies.
*   **Test Arguments:** The test arguments are also modified to limit the scope of the tests:
    *   `--packages-up-to rcpputils`: This limits the tests to `rcpputils` and its dependencies.
*   **Docker Image:** The Docker image is tagged as `ros2_batch_ci`.

## Execution Flow

The execution flow of the `nightly_linux_address_sanitizer` job is similar to the `ci_linux` job, with the main difference being the build environment and the scope of the build and test.

1.  **SCM Checkout:** The job checks out the `ros2/ci` repository.

2.  **Docker Environment:** The job runs inside a Docker container.

    *   **Dockerfile Selection:** The job uses the standard `linux_docker_resources/Dockerfile`.
    *   **Docker Image Build:** A new Docker image is built from `Dockerfile` and tagged as `ros2_batch_ci`.
    *   **Docker Container Run:** The job runs the newly built Docker image, passing the specific build and test arguments via the `CI_ARGS` environment variable.

3.  **`run_ros2_batch.py`:** The `ros2_batch_job` Python module within the container handles the build and test process.

## Configuration and Scripts

*   **Job Generation Script:**
    *   `create_jenkins_job.py`: The script that generates the `nightly_linux_address_sanitizer` job.
*   **Jenkins Job Template:**
    *   `job_templates/ci_job.xml.em`: The template for the CI job.
*   **Build Logic:**
    *   `run_ros2_batch.py`: The entry point for the build process inside the Docker container.
*   **Docker configuration:**
    *   `linux_docker_resources/Dockerfile`: The Dockerfile for Linux builds.

## Parameters

The `nightly_linux_address_sanitizer` job accepts the same parameters as the `ci_linux` job, but it is typically run with the default values since it is triggered automatically.
