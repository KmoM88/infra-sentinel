# ci_linux_coverage Job Analysis

## Overview

The `ci_linux_coverage` job is a manually-triggered Jenkins pipeline for running a ROS 2 build on Linux and generating code coverage reports. It is dynamically generated by the `create_jenkins_job.py` script and is an instance of the generic `ci_job` template.

## Job Generation

The `create_jenkins_job.py` script generates the `ci_linux_coverage` job specifically for the `linux` OS. It's designed to measure the test coverage of a specific set of quality-level packages.

## Coverage-Specific Configuration

The `ci_linux_coverage` job has several key differences from the standard `ci_linux` job:

*   **Job Name:** The job is explicitly named `ci_linux_coverage`.
*   **Enable Coverage:** The `enable_coverage_default` parameter is set to `true`. This enables the collection of coverage data during the build and test process.
*   **Build Type:** The `cmake_build_type` is set to `Debug` to ensure that the generated binaries contain the necessary debugging information for accurate coverage analysis.
*   **Build and Test Arguments:** The job uses a specific set of build and test arguments to focus on a curated list of packages (`quality_level_pkgs` and `testing_pkgs_for_quality_level`). This allows for a more targeted and efficient coverage analysis. It also skips some GUI-related packages.
*   **Test Execution:** The tests are run sequentially (`--executor sequential`) to ensure that the coverage data is collected correctly without interference between tests.

## Execution Flow

The execution flow of the `ci_linux_coverage` job is similar to the standard `ci_linux` job, with the main difference being the build environment and the post-build steps.

1.  **SCM Checkout:** The job checks out the `ros2/ci` repository.

2.  **Docker Environment:** The job runs inside a Docker container.

    *   **Dockerfile Selection:** The job uses the standard `linux_docker_resources/Dockerfile`.
    *   **Docker Image Build:** A new Docker image is built from `Dockerfile` and tagged as `ros2_batch_ci`. The Dockerfile installs `lcov` and a specific version of `lcov-to-cobertura-xml` for coverage report generation.
    *   **Docker Container Run:** The job runs the newly built Docker image, passing the specific build and test arguments via the `CI_ARGS` environment variable. The `ENABLE_COVERAGE=true` environment variable is also set.

3.  **`run_ros2_batch.py`:** The `ros2_batch_job` Python module within the container handles the build process. When coverage is enabled, it performs the following additional steps after the tests have run:
    *   It uses `lcov` to collect the coverage data generated during the build.
    *   It uses `lcov-to-cobertura-xml` to convert the `lcov` data into the Cobertura format, which can be displayed by the Jenkins Cobertura Plugin.

## Configuration and Scripts

*   **Job Generation Script:**
    *   `create_jenkins_job.py`: The script that generates the `ci_linux_coverage` job.
*   **Jenkins Job Template:**
    *   `job_templates/ci_job.xml.em`: The template for the `ci_job`.
*   **Build Logic:**
    *   `run_ros2_batch.py`: The entry point for the build process inside the Docker container.
*   **Docker configuration:**
    *   `linux_docker_resources/Dockerfile`: The Dockerfile for Linux builds.

## Parameters

The `ci_linux_coverage` job accepts the same parameters as the `ci_launcher` and its other downstream jobs, but many of them are overridden with specific values to configure the coverage build.
