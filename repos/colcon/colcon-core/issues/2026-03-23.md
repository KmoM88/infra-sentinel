# Technical Report: colcon/colcon-core Open Issues Analysis (2026-03-23)

## Introduction

This report provides a deep technical analysis of the top 5 most critical open issues in the `colcon/colcon-core` repository. The issues were selected based on their potential impact on developer experience, workflow correctness, and long-term maintainability of the tool. For each issue, a summary of the problem is provided, along with a proposed technical approach for resolution and a list of files that would likely require modification.

---

## 1. Issue #633: Error building after deleting a file with `--symlink-install`

-   **Problem Summary:** When using `colcon build --symlink-install`, if a file included via a wildcard in a `setup.py`'s `data_files` is deleted from the source, the subsequent build fails. This happens because a broken symbolic link remains in the build directory, causing an error when `setuptools` tries to process it. The current workaround is to manually delete the package's build directory.

-   **Technical Approach:** The core of the issue lies in how `setuptools` handles `data_files` during a symlink install. The most robust long-term solution is to transition to a standards-based build backend (PEP 517), which is a significant effort. A more immediate, pragmatic solution would be to have `colcon` clean up these dangling symlinks before the build step. This could be implemented as a pre-build action within the Python build task that scans the build directory for broken symlinks corresponding to `data_files` and removes them.

-   **Files to Modify:**
    -   `colcon_core/task/python/build.py`: To add a pre-build step to clean up dangling symlinks from the build directory.
    -   Alternatively, `colcon_python_setup_py/colcon_python_setup_py/task/python/build.py`: The logic could be added to the specific extension that handles `setup.py` builds.

## 2. Issue #194: Warn user if they `colcon build` after `source install/setup`

-   **Problem Summary:** A frequent user error is to run `colcon build` in a shell that has already sourced an `install/setup.bash` file from the same workspace. This "pollutes" the build environment, potentially leading to incorrect dependency resolution, hidden bugs, and slower builds. The tool does not currently warn the user about this incorrect workflow.

-   **Technical Approach:** A warning mechanism should be implemented to detect this situation. The most reliable method would be to set a "canary" environment variable in the `setup.sh` script (e.g., `COLCON_SOURCED_WORKSPACE`). During the execution of `colcon build`, the tool would check for the presence of this variable. If it exists and points to the current workspace, a prominent warning should be displayed to the user, advising them to use a clean shell for building.

-   **Files to Modify:**
    -   `colcon_core/verb/build.py`: The `main` method of the `BuildVerb` is the ideal location to add the environment check and display the warning.
    -   `colcon_core/shell/template/prefix.sh.em`: The setup script template should be modified to export the canary environment variable.

## 3. Issue #671: Treat `exec_dependencies` differently

-   **Problem Summary:** `colcon` does not currently differentiate between build-time dependencies (`build_depend`, `test_depend`) and run-time dependencies (`exec_depend`) when determining the build order. This limits build parallelization and prevents certain valid software architectures, such as a plugin (`B`) having a `build_depend` on an API package (`A`), while `A` has an `exec_depend` on `B` for testing purposes.

-   **Technical Approach:** This requires a fundamental change to colcon's dependency handling. The dependency resolution logic must be updated to create a build-ordering graph based solely on build-time dependencies, while still ensuring that all packages (including run-time dependencies) are available in the environment where they are needed. This is a complex architectural change that would likely need to be introduced behind a feature flag to avoid breaking existing packages that may have incorrect dependency declarations.

-   **Files to Modify:**
    -   `colcon_ros/package_identification/ros.py`: To preserve the distinction between different dependency types.
    -   `colcon_core/package_selection.py`: To handle the fine-grained dependency information during package selection.
    -   `colcon_core/executor.py`: To use only build-time dependencies for topological sorting of the build order.
    -   `colcon_core/task/__init__.py`: To ensure the task execution environment is correctly populated with the necessary dependencies.

## 4. Issue #139: Support invocation in subfolders of the workspace

-   **Problem Summary:** `colcon` commands must be invoked from the root of a workspace. Running `colcon` from a subdirectory results in the creation of new `build`, `install`, and `log` folders in that subdirectory, which is counter-intuitive and differs from the behavior of other common development tools like `git`.

-   **Technical Approach:** The tool should be made "workspace-aware" by searching upwards from the current directory for a marker file (e.g., `.colcon_workspace`) that identifies the workspace root. Once found, `colcon` should operate as if it were invoked from that root directory. A critical part of this implementation is to correctly handle any relative paths provided as command-line arguments, which must be resolved relative to the original working directory before the process's CWD is changed. A community extension (`colcon-top-level-workspace`) already provides this functionality, and its approach should be considered for integration into the core tool or as an official extension.

-   **Files to Modify:**
    -   `colcon_core/command.py`: The `main` function should be modified to include the workspace discovery logic before argument parsing.
    -   A new module for workspace discovery would be required.
    -   The argument parsing logic may need to be adapted to handle the resolution of relative paths before the CWD is changed.

## 5. Issue #454: `setup.py` deprecation warning

-   **Problem Summary:** Building Python packages with `colcon` produces a `SetuptoolsDeprecationWarning` because it directly invokes `setup.py install`. This practice is deprecated in the Python ecosystem in favor of standards-based build frontends that use `pyproject.toml` and PEP 517. While currently just a warning, it signals a future breaking change in `setuptools`.

-   **Technical Approach:** The ultimate solution is a transition to a modern, PEP 517-compliant build process. This is a large-scale effort that involves creating a new colcon extension (e.g., `colcon-python-build`) to handle `pyproject.toml`-based builds and potentially requires updates to many existing ROS packages. A prototype already exists but has limitations. The long discussion and high engagement on this issue highlight its importance for the long-term health of the ecosystem. A short-term fix to suppress the warning has been implemented, but the underlying issue remains. The path forward is to continue the development and testing of the new standards-based build extension.

-   **Files to Modify (for the full fix):**
    -   This is not a simple file change but a larger architectural shift. It would involve:
        -   The deprecation and eventual replacement of `colcon-python-setup-py`.
        -   The creation and stabilization of a new extension (e.g., `colcon-python-build`) for PEP 517 builds.
        -   Potential modifications in `colcon-core` to accommodate the new build workflow (e.g., handling wheel artifacts).
