# Code Review: PR #78 - Stop using pkg_resources in setup.py

**Date:** 2026-03-23
**Pull Request:** https://github.com/colcon/colcon-notification/pull/78

---

## 1. Contextual Analysis

This pull request is a proactive dependency cleanup. It removes the usage of the deprecated `pkg_resources` library from `setup.py`. The script was using `pkg_resources.parse_version` to check if the installed `setuptools` version met the minimum requirement.

The PR replaces this with a small, self-contained helper function (`_try_parse`) that performs a basic version comparison. This is a necessary maintenance task to ensure the package remains compatible with future versions of Python's packaging toolchain, where `pkg_resources` will be fully removed.

## 2. Technical Evaluation

### Efficiency & Readability
-   **Readability:** The change is clear. The new `_try_parse` function is small and its purpose is easy to understand within the context of the `setup.py` script.
-   **Efficiency:** The custom function is perfectly efficient for this simple version check.

### Security
-   There are no security implications. The change is isolated to the package's setup script.

### Reliability
-   The change improves the long-term reliability of the package by removing a dependency on a deprecated and soon-to-be-removed library.
-   The new version parsing logic is simple and sufficient for its purpose of checking the `setuptools` version number. It correctly handles pre-release tags by splitting the version string.

---

## 3. Critical Inquiries & Proactive Improvements

1.  **Alternative (Using `packaging` library):** This is a great removal of a deprecated dependency. However, instead of a custom parsing function, have you considered using `packaging.version.parse`? The `packaging` library is the modern standard for this, and since `colcon-core` depends on it, it should already be available in the installation environment. This would be more robust and avoid custom parsing logic.
    ```python
    from packaging.version import parse as parse_version
    # ...
    if parse_version(setuptools_version) < parse_version('40.5.0'):
        # ...
    ```

2.  **Readability of Parsing Logic:** The line `raw = next(iter((raw or '').split('-', 1)))` is a clever and concise way to strip pre-release tags. To help future readers who may not be familiar with this pattern, would you consider adding a brief comment explaining its purpose?

3.  **Ecosystem Consistency:** This `pkg_resources` check exists in many of the `colcon` packages. Is there a plan to apply this or a similar fix across the other `colcon` repositories to ensure they are all prepared for the removal of `pkg_resources`?

---

## 4. Recommendation

**Approve.**

This is a clean, correct, and proactive fix that improves the long-term health of the package. While using the standard `packaging` library would be a slight improvement, the current implementation is perfectly acceptable and achieves the goal of removing the deprecated dependency.

-   **Must-Fix:**
    -   None.
-   **Nice-to-Have:**
    -   Consider replacing the custom `_try_parse` function with `packaging.version.parse` for better standardization and robustness.

---

## 5. Approval Justification & Positive Impact

-   **Success Factors:** This PR is a great example of proactive maintenance. It successfully **removes a deprecated dependency (`pkg_resources`)** with a minimal and self-contained change, ensuring future compatibility without adding any new dependencies.
-   **Downstream Benefits:** This change **improves the long-term maintainability** of the package. By removing `pkg_resources`, it ensures that `colcon-notification` will continue to install correctly with future versions of `setuptools`, preventing potential breakage for users in the future.
-   **Risk Assessment:** The risk of regression is extremely low. The change is confined to the `setup.py` script and only affects a version check during installation. The existing CI pipeline will validate that the package remains installable and functional.
