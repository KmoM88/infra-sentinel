
# Technical Analysis of ros-infrastructure/ros_buildfarm

## 1. Repository Discovery & Branching Logic

- **Primary Branch:** The primary branch is `master`.
- **Branching Strategy:** The repository uses a `master`-based workflow. The numerous other branches appear to be for feature development, bug fixes, and testing by various contributors. The `master` branch represents the main line of development.

## 2. Core Purpose & Architecture

- **Core Purpose:** This repository provides the scripts and tools for the modern ROS buildfarm. It is the official replacement for the deprecated `ros-infrastructure/buildfarm` repository. Its purpose is to automate the building, testing, and packaging of ROS 1 and ROS 2 packages into Debian (`.deb`) and RPM (`.rpm`) packages.
- **Architecture:** The architecture is a sophisticated, multi-layered system:
  - **Core Logic:** A Python package (`ros_buildfarm`) that contains the logic for generating build, test, and release jobs.
  - **Configuration:** It consumes configuration from a separate repository, typically `ros-infrastructure/ros_buildfarm_config`, which defines the ROS distributions, repositories, and build targets.
  - **Job Generation:** The tools can generate either Jenkins Groovy scripts for a full-scale CI/CD environment or local shell scripts for development and testing.
  - **Execution Environment:** It heavily utilizes Docker to create isolated and reproducible build environments for every step of the process.

## 3. Consumption (Inputs)

- **External Libraries/Frameworks:** The core Python dependencies are listed in `setup.py`:
  - `empy`: For templating job configurations.
  - `PyYAML`: for parsing configuration files.
  - `catkin_pkg`: For introspecting ROS packages.
  - `jenkinsapi`: For interacting with the Jenkins API.
  - `rosdistro`: For fetching ROS distribution information.
  - `vcstool`: For checking out source code from version control.
- **Other Repositories:** The system is explicitly designed to work with a separate configuration repository (e.g., `ros-infrastructure/ros_buildfarm_config`) and consumes data from countless other ROS package repositories across GitHub.
- **Required APIs:** The primary external API is the Jenkins API, used to create, configure, and trigger jobs.

## 4. Production (Outputs)

- **CI/CD Jobs:** The primary output is a set of configuration files for Jenkins (Groovy scripts) or executable shell scripts that define the entire build process.
- **Packages:** The end-product of the *jobs generated by this repository* are Debian (`.deb`) and RPM (`.rpm`) packages for ROS distributions.
- **PyPI Package:** The `ros_buildfarm` code itself is packaged and can be installed from PyPI.

## 5. CI/CD Pipeline Analysis

The repository uses a hybrid approach for CI/CD:

- **GitHub Actions:** A comprehensive GitHub Actions workflow is defined in `.github/workflows/ci.yaml`.
  - **Triggers:** The workflow is triggered on `push` to the `master` branch and on any `pull_request`.
  - **Stages:** It runs a variety of jobs to test the `ros_buildfarm` scripts themselves, not to build ROS packages directly. These jobs include:
    - `pytest`: Runs the Python unit tests.
    - `ros1_audit`, `ros2_audit`: Simulates running an audit of the ROS distributions.
    - `ros1_release`, `ros2_release`: Simulates running release jobs.
    - `ros1_doc`, `ros2_doc`: Simulates running documentation generation jobs.
    - This workflow ensures that the job generation logic remains correct and functional.
- **Jenkins:** While GitHub Actions tests the generator, the *output* of the tool is designed to be consumed by a Jenkins server. The repository contains Groovy templates (`.groovy`) that are populated with configuration to create the actual buildfarm jobs on a Jenkins instance.

## 6. Standalone Usage Guide

The `README.md` and `doc/index.rst` provide guidance on running the tools. The following is a brief "Quick Start" based on the `README.md`.

1.  **Installation:** Install the package and its dependencies from the root of the repository:
    ```bash
    python3 -m pip install .
    ```

2.  **Run Scripts:** The tools are invoked via scripts in the `scripts` directory. These scripts typically require a URL to a `ros_buildfarm_config` `index.yaml` file and a ROS distribution name.

3.  **Example (Check Sync Criteria for ROS Melodic):**
    ```bash
    ./scripts/release/check_sync_criteria.py https://raw.githubusercontent.com/ros-infrastructure/ros_buildfarm_config/production/index.yaml melodic default ubuntu bionic amd64
    ```

4.  **Example (Audit ROS Noetic):**
    ```bash
    ./scripts/release/audit_rosdistro.py --cache-dir /tmp/rosdistrocache https://raw.githubusercontent.com/ros-infrastructure/ros_buildfarm_config/production/index.yaml noetic
    ```
