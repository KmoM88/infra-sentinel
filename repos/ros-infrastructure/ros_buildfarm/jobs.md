# ROS 2 Buildfarm Job Ecosystem Analysis

This document provides a detailed breakdown of the jobs generated by `ros_buildfarm` based on the configuration in the `ros2/ros_buildfarm_config` repository.

## 1. CI Builds (`ci-build`)

CI (Continuous Integration) jobs are primarily designed for running builds and tests on the latest source code. They serve as the main feedback loop for developers.

### Example: Infrastructure CI (`ci_infra`)

*   **Defining File:** `global/ci-infra.yaml`
*   **Purpose:** This job provides continuous integration for the core ROS infrastructure packages themselves (e.g., `ros_buildfarm`, `rosdep`). It ensures that the tools used to build ROS 2 are stable.
*   **Trigger:** Nightly cron schedule (`10 23 * * *`).
*   **Execution Steps:**
    1.  **Agent:** Runs on a Jenkins agent with the label `ci-agent`.
    2.  **Dependencies:** Installs a list of prerequisite Debian packages.
    3.  **Checkout:** Clones repositories from `ros-infrastructure.repos`.
    4.  **Build & Test:** Uses `colcon` to build the workspace and run tests, skipping specific tests like spell-checking.
*   **Interactions:** Interacts with Git to fetch sources and with Apt repositories to install dependencies.
*   **Docker:** This job does not build a Docker container; it relies on a pre-configured Jenkins agent environment.

## 2. Release Builds (`release-build`)

Release builds are responsible for creating the distributable binary packages (e.g., `.deb` on Debian/Ubuntu) that end-users install. This is a two-stage process.

### Example: Humble Release Build (`Hrel...`)

*   **Defining File:** `humble/release-build.yaml`
*   **Purpose:** To generate Debian binary packages for the ROS 2 Humble distribution. This process creates two types of chained jobs for each ROS package:
    *   **Source Jobs (`_source`):** Generates a source Debian package (`.dsc`) from a tagged, released version of a package.
    *   **Binary Jobs (`_binary`):** Compiles the source package into a binary `.deb` for each target architecture.
*   **Trigger:** Typically triggered manually by a release manager via a `trigger-jobs` maintenance job after a package is released into the `rosdistro` database.
*   **Execution Steps:**
    1.  **Source Job:** Checks out the released source from its git repository and creates the source Debian package.
    2.  **Binary Job:** This step runs inside a Docker container. It downloads the source package, installs build dependencies from the configured Apt repositories, and uses the platform's native toolchain (e.g., `dpkg-buildpackage`, `sbuild`) to create the final `.deb` file.
    3.  **Upload:** The final binary package is uploaded to the specified `target_repository`.
*   **Interactions:** Jenkins (orchestration), `rosdistro` database (package versioning), Git (source fetching), Apt repositories (dependencies and upload target).
*   **Docker:**
    *   **Usage:** Yes, the binary build step is containerized.
    *   **Dockerfile:** The `Dockerfile` is generated **dynamically** by `ros_buildfarm` scripts. The base image is determined by the `targets` configuration (e.g., `ubuntu:jammy` for this job).

## 3. Source (Devel) Builds (`source-build`)

Source builds, also known as "devel" jobs, continuously build and test the main development branches (e.g., `rolling`, `humble`) of packages. They provide rapid feedback on the latest changes.

### Example: Humble Source Build (`Hdev...`)

*   **Defining File:** `humble/source-build.yaml`
*   **Purpose:** To continuously build the `HEAD` of development branches for ROS 2 Humble packages. It identifies integration issues and regressions as soon as they are committed.
*   **Trigger:** SCM Polling. The configuration `test_commits: true` enables jobs that are triggered on every new commit pushed to a tracked repository branch.
*   **Execution Steps:**
    1.  **Checkout:** Checks out the latest source of the changed repository and its dependencies using `vcstool`.
    2.  **Build & Test:** Uses `colcon` to build the workspace and run tests.
    3.  **Notifications:** Sends email notifications to maintainers and committers on build failures.
*   **Interactions:** Jenkins, GitHub (polling for commits), `rosdistro` (dependency resolution), Apt repositories.
*   **Docker:**
    *   **Usage:** Yes, the entire process runs inside a Docker container.
    *   **Dockerfile:** Generated **dynamically** by `ros_buildfarm` scripts based on the target OS specified in the YAML file (e.g., `ubuntu:jammy`).

## 4. Documentation Builds (`doc-build`)

Documentation builds are responsible for generating and deploying the API documentation for ROS packages.

### Example: Independent Doc Build

*   **Defining File:** `global/doc-independent-pkgs-build.yaml`
*   **Purpose:** To generate and publish documentation for core, distribution-agnostic packages like `rosdep` and `rospkg`.
*   **Trigger:** Scheduled (nightly) or manual.
*   **Execution Steps:**
    1.  **Agent:** Runs on a Jenkins agent.
    2.  **Dependencies:** Installs `sphinx` and other documentation tools via `apt` and `pip`.
    3.  **Checkout:** Clones the specific repositories to be documented.
    4.  **Generate Docs:** Runs a `make` target within each repository to build the Sphinx documentation.
    5.  **Upload:** Uploads the generated HTML via SSH to the target web server (`docs.ros.org`).
*   **Interactions:** Git (source fetching), Apt/Pip (tool installation), SSH (deployment).
*   **Docker:** This job does not build a Docker container; it runs on a pre-configured agent.
