# Technical Analysis of `osrf/buildfarm-tools/scripts/`

## 1. High-Level Overview

The `scripts/` directory within the `osrf/buildfarm-tools` repository contains a collection of utility scripts written in GNU Awk (`gawk`). These scripts are designed for parsing and extracting specific information from Jenkins build logs generated by the ROS and Gazebo buildfarms. Their primary function is to automate common tasks related to build log analysis, such as fetching logs, isolating package-specific output, and extracting version control information.

These scripts are lightweight, efficient, and designed to be used in a command-line environment, making them suitable for integration into larger shell scripts and automation workflows.

---

## 2. Granular Breakdown

### `ci-log`

- **Responsibility:** This script is responsible for fetching the full console log of a Jenkins build and saving it to a local file.
- **Logic & Data Flow:**
    1.  It accepts a single argument: the URL of a Jenkins build job (e.g., `https://ci.ros2.org/view/nightly/job/nightly_linux-aarch64_debug/1641/`).
    2.  A regular expression is used to parse the URL and extract the Jenkins server address, job name, and build number.
    3.  It constructs the URL for the raw console log (`/consoleText`).
    4.  It checks for the existence of `JENKINS_USERNAME` and `JENKINS_API_TOKEN` environment variables. If they are set, it includes them in the `curl` command for basic authentication.
    5.  It executes a `curl` command to download the log content.
    6.  The log is saved to a file named `{job_name}-{build_number}.txt`.
- **Key Functions:** The entire logic is contained within the `BEGIN` block of the `gawk` script.

### `extract-package-output`

- **Responsibility:** This script extracts the build or test output for a single package from a larger build log, using the structured output markers from `colcon`.
- **Logic & Data Flow:**
    1.  It takes three arguments: `OPERATION` (`build` or `test`), `PACKAGE` (the name of the package), and an optional `BUILD_LOG_FILE`. If the file is not provided, it reads from standard input.
    2.  It identifies the start of the relevant section using a `section_pattern` (e.g., `BEGIN SUBSECTION: build`).
    3.  Within that section, it waits for the `start_pattern` (e.g., `Starting >>> my_package`).
    4.  Once the start pattern is found, it begins printing every line.
    5.  It stops printing and exits when it encounters the `finish_pattern` (e.g., `Finished <<< my_package` or `Failed <<< my_package`).
- **Key Functions:** The script uses state variables (`in_section`, `start`) to track its position within the log file and determine when to print lines.

### `extract-vcs-exact`

- **Responsibility:** This script extracts the output of the `vcs export --exact` command from a build log. This output is a YAML-based `.repos` file format that specifies the exact commit hashes of all repositories used in a build.
- **Logic & Data Flow:**
    1.  It reads from a build log provided via standard input or as a file argument.
    2.  It searches for the line `# BEGIN SUBSECTION: vcs export --exact` to start the extraction.
    3.  It then prints every subsequent line.
    4.  It includes logic to skip common noise, such as the `(venv)` shell prefix and the `==> vcs export...` command echo.
    5.  The script terminates when it encounters either `# END SUBSECTION` or an empty line, which typically signifies the end of the `vcs` output.
- **Key Functions:** A state variable `in_vcs_exact` is used to control when lines should be printed to the output.

## 3. Usage & Implementation

These scripts are intended to be used from the command line, often piped together with other shell commands.

### `ci-log`

**Example:** Fetch the log for a specific Jenkins build.

```bash
# Fetch the log for build #1641 of the nightly_linux-aarch64_debug job
./scripts/ci-log https://ci.ros2.org/view/nightly/job/nightly_linux-aarch64_debug/1641/

# This will create a file named nightly_linux-aarch64_debug-1641.txt
```

### `extract-package-output`

**Example:** Extract the test output for the `rclcpp` package from a downloaded log file.

```bash
# First, get the log file
./scripts/ci-log <JENKINS_BUILD_URL> > build.log

# Then, extract the test output for rclcpp
./scripts/extract-package-output test rclcpp build.log
```

### `extract-vcs-exact`

**Example:** Generate a `.repos` file from a build log to reproduce a build's sources.

```bash
# First, get the log file
./scripts/ci-log <JENKINS_BUILD_URL> > build.log

# Then, extract the vcs information and save it to a file
./scripts/extract-vcs-exact build.log > my_build.repos

# You can now use this file with the vcs tool
# vcs import src < my_build.repos
```

## 4. Technical Nuances

- **Dependencies:**
    - `gawk` (GNU Awk): The shebang `#!/usr/bin/gawk -f` specifies that these scripts must be run with GNU Awk. They may not be compatible with other Awk implementations due to the use of GNU-specific features (like `ARGC` manipulation).
    - `curl`: The `ci-log` script has an external dependency on the `curl` command-line utility.
- **Performance:** As `gawk` scripts, they are extremely efficient at text processing. They stream through log files line-by-line, consuming very little memory, which makes them suitable for analyzing very large log files (often hundreds of megabytes).
- **Edge Cases & Robustness:**
    - The `ci-log` script includes a basic check for Jenkins authentication credentials in environment variables, making it adaptable to different Jenkins setups.
    - The regular expression for URL parsing in `ci-log` is specific to the common URL structures of the ROS/Gazebo buildfarms and might fail on differently structured Jenkins URLs.
    - The output extraction scripts (`extract-package-output`, `extract-vcs-exact`) are tightly coupled to the specific log format produced by `colcon` and the buildfarm infrastructure. Changes to the log format could break these scripts.
    - Error handling is minimal and relies on printing messages to `stderr` and exiting with a non-zero status code, which is standard practice for command-line utilities.
